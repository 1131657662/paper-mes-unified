# 纸品加工 MES 生产部署指南

本文按个人开发者的一台云服务器部署来写，目标是够用、稳、容易恢复，不引入复杂运维平台。

## 推荐架构

- Nginx 对外监听 `80/443`，托管前端 `dist`。
- Nginx 将 `/api/` 和 `/files/` 反向代理到后端 `127.0.0.1:8081`，让上传文件也经过后端登录鉴权。
- Spring Boot 后端用 systemd 守护，不直接暴露公网端口。
- MySQL 只给本机应用账号访问，生产不要使用 `root` 连接业务系统。
- 上传文件目录固定为 `/opt/paper-mes/upload`，和数据库一起备份。
- 前端图片预览走 `/api/files/**` 受保护接口；历史 `/files/**` 路径只作兼容，不要在 Nginx 里直接 `alias` 到上传目录。

## 目录约定

```bash
/opt/paper-mes/app/paper-mes.jar
/opt/paper-mes/frontend/dist
/opt/paper-mes/upload
/etc/paper-mes/application-prod.yml

```

## 数据库账号

示例命令如下，请把密码替换为强密码：

```sql
CREATE DATABASE IF NOT EXISTS paper_processing
  DEFAULT CHARACTER SET utf8mb4
  DEFAULT COLLATE utf8mb4_0900_ai_ci;

CREATE USER IF NOT EXISTS 'paper_mes_app'@'127.0.0.1'
  IDENTIFIED BY 'CHANGE_ME_STRONG_PASSWORD';
GRANT SELECT, INSERT, UPDATE, DELETE
  ON paper_processing.* TO 'paper_mes_app'@'127.0.0.1';

CREATE USER IF NOT EXISTS 'paper_mes_migrator'@'127.0.0.1'
  IDENTIFIED BY 'CHANGE_ME_MIGRATION_PASSWORD';
GRANT SELECT, INSERT, UPDATE, DELETE, CREATE, ALTER, INDEX, REFERENCES, DROP
  ON paper_processing.* TO 'paper_mes_migrator'@'127.0.0.1';

CREATE USER IF NOT EXISTS 'paper_mes_backup'@'127.0.0.1'
  IDENTIFIED BY 'CHANGE_ME_BACKUP_PASSWORD';
GRANT SELECT, LOCK TABLES, SHOW VIEW, EVENT, TRIGGER
  ON paper_processing.* TO 'paper_mes_backup'@'127.0.0.1';
FLUSH PRIVILEGES;
```

初始化表结构：

```bash
mysql -u root -p < sql/00_init.sql
mysql -u root -p paper_processing < sql/01_schema_v4.1.sql
mysql -u root -p paper_processing < sql/02_index_p3-3.sql
```

历史变更脚本不要再用 `cat sql/V*.sql` 手动拼接执行。先复制迁移配置，再由迁移脚本按版本顺序执行并记录 checksum：

```bash
sudo cp deploy/migration.env.example /etc/paper-mes/migration.env
sudo chmod 600 /etc/paper-mes/migration.env
sudo cp deploy/apply-paper-mes-migrations.example.sh /usr/local/bin/apply-paper-mes-migrations
sudo chmod 700 /usr/local/bin/apply-paper-mes-migrations
```

编辑 `/etc/paper-mes/migration.env`：

- `DB_USER` 使用独立的 `paper_mes_migrator` 迁移账号，不要使用应用账号。
- `DB_PASSWORD` 使用真实强密码。
- `MIGRATION_DIR` 指向服务器上的项目 `sql` 目录，例如 `/opt/paper-mes/source/sql`。

全新数据库先执行当前 `01_schema_v4.1.sql` 和 `02_index_p3-3.sql`。当前基线已经包含全部历史迁移后的结构，因此不要在新库上重放旧 `ALTER` 脚本；应先登记 baseline：

```bash
sudo MIGRATION_BASELINE=1 /usr/local/bin/apply-paper-mes-migrations
```

baseline 登记完成后，未来新增版本再使用普通模式执行。已有数据库如果已经维护 `sys_schema_migration`，每次发布前先备份，再直接运行普通模式：

```bash
sudo /usr/local/bin/apply-paper-mes-migrations
```

脚本会记录 `V*.sql`、文件名、checksum 和执行时间；已经执行过且 checksum 一致的脚本会跳过；如果已执行脚本被改动导致 checksum 不一致，会拒绝继续，避免生产库和代码库悄悄跑偏。

如果已有生产库从未维护迁移历史，且历史 `V*.sql` 大多已经通过手工或 bootstrap 补过，不要直接重跑所有历史脚本。先备份并确认当前表结构已经与当前代码对齐，再做一次 baseline 登记：

```bash
sudo MIGRATION_BASELINE=1 /usr/local/bin/apply-paper-mes-migrations
```

baseline 只把当前 `V*.sql` 记录到 `sys_schema_migration`，不执行 SQL。之后新增的 `V2.8`、`V2.9` 等脚本再由普通模式实际执行。`V2.7__fix_process_config_draft_version_default.sql` 会把 `biz_process_config_draft.version` 默认值统一为 `1`，并修正历史 `version=0` 的草稿数据；如果生产库还没应用这项修复，先单独执行该脚本或启动新版后端让 bootstrap 修复，再做 baseline。

生产配置默认关闭运行时建表和改字段，所有 DDL 必须先由迁移账号执行。`paper_mes_app` 始终只保留 `SELECT/INSERT/UPDATE/DELETE`；默认配置、单号规则和空库初始用户仍由应用以普通 DML 初始化。启动前必须确认迁移已经执行到最新版本，否则后端应视为不可发布，而不是临时给应用账号补 DDL 权限。

## 后端配置

从模板复制一份真实配置到服务器：

```bash
sudo mkdir -p /etc/paper-mes
sudo cp src/main/resources/application-prod.example.yml /etc/paper-mes/application-prod.yml
sudo cp deploy/paper-mes.env.example /etc/paper-mes/paper-mes.env
sudo chmod 600 /etc/paper-mes/application-prod.yml
sudo chmod 600 /etc/paper-mes/paper-mes.env
```

编辑 `/etc/paper-mes/application-prod.yml`：

- `spring.datasource.username` 使用 `paper_mes_app`。
- `PAPER_MES_DB_PASSWORD` 在 `/etc/paper-mes/paper-mes.env` 中使用应用账号真实强密码。
- `app.schema-bootstrap.enabled` 必须保持 `false`，生产结构只允许迁移账号变更。
- `app.upload.dir` 保持 `/opt/paper-mes/upload`。
- 空库首次上线时，必须配置 `app.initial.admin-password` 和 `app.initial.operator-password` 为 8–32 位且包含字母和数字的强密码；生产环境缺少该配置会拒绝启动。
- `server.address` 保持 `127.0.0.1`，让后端只接受本机 Nginx 转发。

## 构建与发布

后端：

```bash
./mvnw.cmd clean package
```

服务器上放置 Jar：

```bash
sudo mkdir -p /opt/paper-mes/app /var/log/paper-mes /opt/paper-mes/upload
sudo cp target/paper-mes-0.0.1-SNAPSHOT.jar /opt/paper-mes/app/paper-mes.jar
```

前端：

```bash
cd frontend
npm ci
npm run build
```

把 `frontend/dist` 发布到：

```bash
/opt/paper-mes/frontend/dist
```

## systemd

复制服务模板：

```bash
sudo useradd --system --home /opt/paper-mes --shell /usr/sbin/nologin paper-mes
sudo chown -R paper-mes:paper-mes /opt/paper-mes /var/log/paper-mes
sudo cp deploy/paper-mes.service.example /etc/systemd/system/paper-mes.service
sudo systemctl daemon-reload
sudo systemctl enable --now paper-mes
```

常用检查：

```bash
systemctl status paper-mes
journalctl -u paper-mes -n 200 --no-pager
curl --fail http://127.0.0.1:8081/actuator/health
```

健康接口应返回 HTTP `200`，响应中的 `status` 必须为 `UP`；启用存活与就绪探针后还会包含安全的分组名称。生产配置不展示数据库、磁盘或配置详情；后端只监听 `127.0.0.1:8081`，Nginx 也不代理 `/actuator/**`，因此该接口不会直接暴露到公网。

## 健康监控与告警

安装五分钟巡检任务：

```bash
sudo install -o root -g root -m 0755 \
  deploy/monitor-paper-mes.example.sh /usr/local/bin/monitor-paper-mes
sudo install -o root -g paper-mes -m 0640 \
  deploy/monitor.env.example /etc/paper-mes/monitor.env
sudo cp deploy/paper-mes-monitor.service.example /etc/systemd/system/paper-mes-monitor.service
sudo cp deploy/paper-mes-monitor.timer.example /etc/systemd/system/paper-mes-monitor.timer
sudo systemctl daemon-reload
sudo systemctl enable --now paper-mes-monitor.timer
```

巡检同时检查：

- 后端 `/actuator/health` 是否为 `UP`
- 可选的公网 HTTPS 地址是否可访问
- 最近一份备份是否超过允许时效、是否存在 `SHA256SUMS`
- 备份磁盘可用空间是否低于阈值

编辑 `/etc/paper-mes/monitor.env` 可填写 `PUBLIC_URL`、备份时效、磁盘阈值和 Webhook。Webhook 地址及 Bearer Token 只能保存在该文件，不能写入仓库、systemd 单元或命令历史。巡检只在状态由正常转为故障、或由故障恢复时发送消息；如果 Webhook 临时不可用，会保留待发送状态并在下次巡检重试。

安装后立即验证：

```bash
sudo systemctl start paper-mes-monitor.service
systemctl status paper-mes-monitor.service --no-pager
systemctl list-timers paper-mes-monitor.timer --no-pager
journalctl -u paper-mes-monitor.service -n 50 --no-pager
```

上线前应做一次不停止业务服务的可控故障演练：临时把 `HEALTH_URL` 改到未监听的本机端口，手动启动巡检并确认收到 `FAILED`；恢复正确地址后再次启动，确认收到 `RECOVERED`。演练结束必须确认 `/var/lib/paper-mes/monitor.state` 为 `UP`，且 `/etc/paper-mes/monitor.env` 已恢复正式配置。

## 日志切割

systemd 服务会把后端标准输出写到 `/var/log/paper-mes/stdout.log`，标准错误写到 `/var/log/paper-mes/stderr.log`；备份 cron 建议写到 `/var/log/paper-mes/backup.log`。这些日志需要定期切割，避免长期增长占满磁盘。

安装模板：

```bash
sudo cp deploy/logrotate-paper-mes.example /etc/logrotate.d/paper-mes
sudo chmod 644 /etc/logrotate.d/paper-mes
sudo logrotate -d /etc/logrotate.d/paper-mes
```

`-d` 是调试模式，只检查配置，不会真正切割日志。确认无误后可手动强制跑一次：

```bash
sudo logrotate -f /etc/logrotate.d/paper-mes
ls -lh /var/log/paper-mes
```

模板只匹配 `/var/log/paper-mes/*.log`，不会影响同服务器上的其它项目。它使用 `copytruncate`，所以不需要重启 `paper-mes` 服务。

## Nginx

复制站点模板：

```bash
sudo cp deploy/nginx-paper-mes.example.conf /etc/nginx/sites-available/paper-mes.conf
sudo ln -s /etc/nginx/sites-available/paper-mes.conf /etc/nginx/sites-enabled/paper-mes.conf
sudo nginx -t
sudo systemctl reload nginx
```

把模板里的 `server_name mes.example.com` 改成你的域名；如果暂时没有域名，可以先写服务器公网 IP。

模板只对本 MES 站点的 `/api/auth/login` 做限流，不会影响同一台服务器上的其它站点。模板顶部的 `limit_req_zone` 需要位于 Nginx `http {}` 上下文；Debian/Ubuntu 默认 `sites-enabled` 就是在 `http {}` 中 include，所以可直接使用。如果你的 Nginx 结构不同，把这一行移到主配置的 `http {}` 内。

模板已启用正式 `Content-Security-Policy`，会阻断不符合策略的资源加载。发布前应在预发布环境验证页面资源、打印预览和文件查看功能。

正式环境必须启用 HTTPS。先用 HTTP 模板完成域名解析和证书签发，再切换到仓库提供的 HTTPS 模板：

```bash
sudo apt-get install -y certbot
sudo certbot certonly --webroot -w /opt/paper-mes/frontend/dist -d mes.example.com
sudo mkdir -p /etc/nginx/snippets
sudo cp deploy/nginx-paper-mes-proxy.example.conf /etc/nginx/snippets/paper-mes-proxy.conf
sudo cp deploy/nginx-paper-mes-https.example.conf /etc/nginx/sites-available/paper-mes.conf
sudo sed -i 's/mes\.example\.com/mes.nbsmzwl.cn/g' /etc/nginx/sites-available/paper-mes.conf
sudo nginx -t
sudo systemctl reload nginx
sudo certbot renew --dry-run
```

模板会把 HTTP 永久跳转到 HTTPS，限制为 TLS 1.2/1.3，并在 HTTPS 响应中启用 HSTS。首次确认 HTTPS、打印、文件预览均正常后再保留 HSTS；云安全组只开放 `22/80/443`，MySQL 和 `8081` 不对公网开放。

## 慢查询与索引排查

不要在没有证据的情况下给生产库盲目加索引。索引会提升部分读取速度，但也会增加写入、回录、出库、结算更新时的维护成本。当前项目已经有一批基础索引，后续新增索引应由慢查询日志或 `EXPLAIN` 结果驱动。

安装只读诊断脚本：

```bash
sudo cp deploy/diagnose-paper-mes-slow-queries.example.sh /usr/local/bin/diagnose-paper-mes-slow-queries
sudo chmod 700 /usr/local/bin/diagnose-paper-mes-slow-queries
```

脚本默认读取 `/etc/paper-mes/migration.env` 中的数据库连接，只执行 `SHOW INDEX`、`information_schema` 查询和 `EXPLAIN FORMAT=JSON SELECT`，不会修改表结构或业务数据：

```bash
sudo /usr/local/bin/diagnose-paper-mes-slow-queries > /tmp/paper-mes-query-plan.txt
```

需要复现特定筛选条件时，可以临时指定参数：

```bash
sudo env CUSTOMER_UUID='客户UUID' ORDER_STATUS=4 DATE_FROM=2026-07-01 DATE_TO=2026-07-31 \
  /usr/local/bin/diagnose-paper-mes-slow-queries > /tmp/paper-mes-query-plan.txt
```

如果是搜索框卡顿，再带上 `KEYWORD`：

```bash
sudo env KEYWORD='客户名或单号' /usr/local/bin/diagnose-paper-mes-slow-queries
```

线上排查建议流程：

1. 先确认是哪一个页面、什么筛选条件、数据量多大时慢。
2. 短时间开启 MySQL 慢查询日志，或用上面的诊断脚本拿到对应查询的 `EXPLAIN`。
3. 只有当结果显示大表全表扫描、额外排序、扫描行数明显过高时，才新增 `sql/Vx.x__add_xxx_index.sql` 迁移脚本。
4. 新索引遵循“等值条件在前、范围条件其次、排序字段最后”的顺序；例如客户 + 状态 + 日期范围 + 创建时间排序这类组合，不能只凭感觉加单列索引。
5. 发布索引前先备份数据库，再通过 `/usr/local/bin/apply-paper-mes-migrations` 执行；不要修改已经执行过的历史迁移脚本。

如果需要更精确的实际执行耗时，可以在业务低峰期手工使用 `EXPLAIN ANALYZE`。它会真实执行查询，不要对写入语句使用，也不要在高峰期随意跑大范围查询。

## 备份

备份必须同时包含数据库和上传文件：

```bash
sudo mkdir -p /opt/backups/paper-mes
sudo cp deploy/backup.env.example /etc/paper-mes/backup.env
sudo chmod 600 /etc/paper-mes/backup.env
```

编辑 `/etc/paper-mes/backup.env`，把 `DB_PASSWORD` 换成 `paper_mes_backup` 的真实密码。该文件不需要 `export`，脚本会自动读取并导出给 `mysqldump`。

管理页面会自动识别运行系统：Windows 使用仓库内的 PowerShell 脚本和项目根目录 `backup/`，Ubuntu 使用仓库内的 Bash 脚本和 `/opt/backups/paper-mes`。云端在 `/opt/paper-mes/source` Pull 新代码后不需要修改脚本路径或后端配置。

手动执行示例：

```bash
sudo BACKUP_ENV_FILE=/etc/paper-mes/backup.env \
  /opt/paper-mes/source/deploy/backup-paper-mes.example.sh
```

数据库备份由后端统一调度，默认每天 `02:35` 执行，可在“系统管理 → 数据安全”调整开关和执行时间。后端会识别当天已有的有效备份，因此兼容升级期间短暂保留的旧 `02:30` cron；确认新版本自动任务成功后，应移除旧数据库备份 cron，避免长期保留两套调度来源。

恢复演练示例，默认导入到 `paper_mes_restore_check` 测试库，验证完成后删除测试库，不覆盖生产库：

```bash
sudo cp deploy/verify-backup-restore.example.sh /usr/local/bin/verify-paper-mes-backup
sudo chmod 700 /usr/local/bin/verify-paper-mes-backup
sudo BACKUP_DIR=/opt/backups/paper-mes/20260707-023000 \
  /usr/local/bin/verify-paper-mes-backup
```

确认输出里 `tables` 大于 0，且 `upload.tar.gz` 没有校验错误。数据库密码只放进 `/etc/paper-mes/backup.env`，不要写进仓库或命令历史。

受控恢复脚本默认拒绝恢复到生产库。优先恢复到新数据库核对；生产恢复必须停写、再次备份当前库，并同时提供目标库确认值和精确的生产确认短语：

```bash
sudo cp deploy/restore-paper-mes.example.sh /usr/local/bin/restore-paper-mes-backup
sudo chmod 700 /usr/local/bin/restore-paper-mes-backup
sudo BACKUP_DIR=/opt/backups/paper-mes/20260707-023000 \
  TARGET_DB_NAME=paper_mes_recovery_review \
  CONFIRM_TARGET_DB_NAME=paper_mes_recovery_review \
  /usr/local/bin/restore-paper-mes-backup
```

受控恢复不会直接覆盖目标库。脚本会先导入到默认名为 `<目标库>_restore_preflight` 的预恢复库，确认 `biz_process_order` 等核心结构可以读取后才处理目标库，并在结束或失败时清理预恢复库。特殊环境可用 `RESTORE_PREFLIGHT_DB_NAME` 指定其他隔离库名，但不得与目标库相同。生产恢复前仍必须保留当前生产库的独立安全备份；预恢复只能证明备份包可导入，不能替代恢复前备份和停写窗口。

异地副本使用独立的 `rclone copy`，远端凭据只保存在 `root` 可读的 rclone 配置中。脚本使用 `copy` 而不是 `sync`，本地保留期清理不会同步删除远端历史备份：

```bash
sudo cp deploy/sync-backups-rclone.example.sh /usr/local/bin/sync-paper-mes-backups
sudo chmod 700 /usr/local/bin/sync-paper-mes-backups
sudo /usr/local/bin/sync-paper-mes-backups
```

同步脚本会在备份根目录原子更新 `.remote-sync-status`，仅记录成功/失败、UTC 完成时间和远端名称。管理端据此展示最近一次异地同步状态；状态文件不包含 rclone 路径、凭据或错误详情，且不会被复制到远端。

开发环境没有真实远端凭据时，可执行 `bash deploy/test-sync-backups-rclone.sh` 验证成功、失败和状态文件更新逻辑。该模拟测试不能代替真实远端验收；正式上线仍必须在远端存储中核对备份目录、文件数量和校验值，并从异地副本完成一次隔离恢复演练。

可直接参考 `deploy/paper-mes-backup.cron.example` 安装每日异地复制和每周恢复演练。数据库备份本身不再写入 cron。

系统管理的“数据安全”页仅允许管理员查看记录、管理自动备份、发起手动备份和隔离恢复演练，不提供网页覆盖生产库。自动备份默认每天 `02:35` 执行，页面展示上次结果、下次执行时间和连续失败次数。默认保留期为 30 天，后端每天 03:15 自动清理过期本地备份；管理员可调整为 7 至 3650 天，也可立即清理或手动删除指定备份。清理优先保护最新一份恢复演练通过的备份；尚无演练通过记录时，至少保护最后一份同时包含数据库压缩包和 `SHA256SUMS` 的完整备份。最后一份恢复验证通过的备份禁止删除。备份脚本不再自行执行固定天数清理，避免与管理端策略冲突。

`PAPER_MES_BACKUP_CLEANUP_CRON` 可在特殊环境覆盖自动清理时间。`PAPER_MES_BACKUP_ROOT`、`PAPER_MES_BACKUP_SCRIPT`、`PAPER_MES_VERIFY_SCRIPT` 仅用于特殊目录覆盖，常规 Windows 和 Ubuntu 部署不需要设置。

## 数据库变更与迁移前数据检查

本版本包含三项增量迁移：

- `V2.8`：备份任务、站内通知、成品剩余重量、待出库库存占用约束和母卷机台索引。
- `V2.9`：客户、纸张、机台和仓库启用状态下的编码唯一约束。
- `V3.0`：加工单按客户分页和创建时间排序的组合索引。

迁移不会替业务人员自动删除重复单据。发布前先执行以下只读检查，所有查询都应返回 0 行：

```sql
-- 同一成品被多张待确认出库单占用。
SELECT d.finish_uuid, COUNT(*) AS lock_count
FROM biz_delivery_detail d
JOIN biz_delivery_order o ON o.uuid = d.delivery_uuid AND o.is_deleted = 0
WHERE d.is_deleted = 0 AND o.delivery_status = 1
  AND d.finish_uuid IS NOT NULL AND TRIM(d.finish_uuid) <> ''
GROUP BY d.finish_uuid
HAVING COUNT(*) > 1;

-- 启用中的基础档案编码重复；四张表分别检查。
SELECT TRIM(customer_code) AS duplicate_code, COUNT(*) AS row_count
FROM sys_customer WHERE is_deleted = 0 AND TRIM(customer_code) <> ''
GROUP BY TRIM(customer_code) HAVING COUNT(*) > 1;
SELECT TRIM(paper_code) AS duplicate_code, COUNT(*) AS row_count
FROM sys_paper WHERE is_deleted = 0 AND TRIM(paper_code) <> ''
GROUP BY TRIM(paper_code) HAVING COUNT(*) > 1;
SELECT TRIM(machine_code) AS duplicate_code, COUNT(*) AS row_count
FROM sys_machine WHERE is_deleted = 0 AND TRIM(machine_code) <> ''
GROUP BY TRIM(machine_code) HAVING COUNT(*) > 1;
SELECT TRIM(warehouse_code) AS duplicate_code, COUNT(*) AS row_count
FROM sys_warehouse WHERE is_deleted = 0 AND TRIM(warehouse_code) <> ''
GROUP BY TRIM(warehouse_code) HAVING COUNT(*) > 1;
```

发现重复占用必须先停止发布，通过出库改单/取消或经审核的数据修复解除冲突并留下操作日志；禁止直接删除明细。发现基础编码重复时，由业务负责人确认应保留的有效档案，再通过正常档案维护流程处理。处理完成后重新备份、复查并执行迁移。

本地可使用 `deploy/test-runtime-migration-conflict.ps1` 在固定隔离库中验证冲突保护。脚本要求通过 `PAPER_MES_IT_DB_PASSWORD` 注入密码，拒绝覆盖已存在的测试库，并在结束后删除自己创建的隔离库。

生产发布前安装并执行统一只读预检：

```bash
sudo install -o root -g root -m 0755 \
  deploy/preflight-paper-mes-release.example.sh /usr/local/bin/preflight-paper-mes-release
sudo /usr/local/bin/preflight-paper-mes-release
```

预检读取现有备份和迁移环境文件，检查后端健康、最新备份时效与 SHA256、运行中备份任务、重复待出库占用和重复启用档案编码。脚本只执行 `SELECT` 和文件校验，任一异常都会以非零状态停止发布。

本地发布前还应执行 `deploy/test-production-profile-startup.ps1`。它使用固定隔离库和独立端口启动生产配置，验证 loopback 监听、健康信息最小暴露、非健康 Actuator 端点不可访问、未登录鉴权和安全 Cookie；测试结束后停止实例并删除自己创建的隔离库。

## 发布顺序与回滚

正式发布按以下顺序执行，任何一步失败都停止后续操作：

1. 确认监控、最近备份和恢复演练状态正常，记录当前 Git 版本、Jar 版本和前端目录。
2. 创建发布前数据库及上传文件备份，并验证 `SHA256SUMS`；保留当前 Jar 和前端构建产物作为回滚副本。
3. 构建后端和前端，完成测试、Lint、生产构建及配置检查，不能在服务器上边改代码边发布。
4. 使用迁移执行器应用尚未执行的 `V*.sql`，核对 `sys_schema_migration`；迁移失败时不要启动新后端。
5. 停止旧后端，原子替换 Jar，启动新后端；只有 `/actuator/health` 返回 `UP` 后才发布前端。
6. 原子替换前端目录，执行登录、加工单详情、打印、出库、结算、备份页冒烟测试，再检查应用日志和监控。

应用回滚只回退 Jar 和前端产物，不应直接删除迁移记录或反向修改数据库。若新版本已写入不兼容数据，应先停写并按审核过的数据回滚方案处理；只有确认必须恢复数据库时，才使用发布前备份和受控恢复脚本。回滚完成后同样需要检查 `/actuator/health`、核心页面、日志、定时备份和监控告警。

## 上线检查清单

- 登录页不展示默认账号密码。
- 默认管理员账号不使用固定初始密码；空库生产初始化前已配置 `app.initial.admin-password` 和 `app.initial.operator-password`。
- `/etc/paper-mes/application-prod.yml` 权限为 `600`，且不在仓库中。
- `/etc/paper-mes/paper-mes.env` 权限为 `600`，数据库与初始账号密码不在仓库中。
- `/etc/paper-mes/migration.env` 权限为 `600`，真实数据库密码不在仓库中。
- 发布前已备份数据库，并执行 `/usr/local/bin/apply-paper-mes-migrations`；`sys_schema_migration` 中能看到最新 `V*.sql` 版本。
- 后端只监听 `127.0.0.1:8081`。
- `/actuator/health` 在服务器本机返回 `UP`，不展示组件详情，且公网无法直接访问 `/actuator/**`。
- `paper-mes-monitor.timer` 已启用；故障与恢复演练均收到告警，最终状态为 `UP`。
- Nginx 可访问前端，`/api/`、`/files/` 能转发到后端；未登录访问上传文件应返回 `401`。
- `/api/auth/login` 限流只在 MES 站点生效，压测触发时返回 `429`。
- 正式 `Content-Security-Policy` 已启用，浏览器控制台无策略拦截错误。
- `/etc/logrotate.d/paper-mes` 已安装并通过 `logrotate -d` 检查。
- MySQL 公网端口未开放；`paper_mes_app` 不是 `root` 且只有 `SELECT/INSERT/UPDATE/DELETE`，DDL 仅由 `paper_mes_migrator` 在发布时执行。
- `app.schema-bootstrap.enabled=false`，应用使用最小权限账号可以正常启动。
- HTTP 自动跳转 HTTPS，TLS 只允许 1.2/1.3，HSTS 生效，`certbot renew --dry-run` 通过。
- 如遇列表卡顿，先用慢查询日志或 `diagnose-paper-mes-slow-queries` 获取证据，再决定是否新增索引迁移。
- `process.weightTolerancePercent` 和 `process.weightBlockTolerancePercent` 已按你的业务容差设置。
- 至少完成一次数据库备份和恢复演练。
- 已保存当前 Jar、前端产物和版本号；发布顺序、应用回滚与数据库恢复责任人已确认。
- 云服务器文档中出现过的密码、Token、默认口令，上线前统一轮换。
