# 纸品加工 MES 投产修复清单

**日期**：2026-07-07
**适用场景**：个人开发者、单台京东云服务器、`mes.nbsmzwl.cn` 公网生产部署
**目标**：先保证生产安全、数据可恢复、线上错误可控，再做性能和体验优化。

---

## 1. 等级标准

| 等级 | 标准 |
|---|---|
| P0 立即修复 | 生产安全、凭据泄露、数据恢复底线。不做不建议继续投产使用。 |
| P1 投产必做 | 明显降低线上 500、安全风险和维护成本。可分批做，但应尽快进入近期计划。 |
| P2 近期优化 | 数据量、使用频率或用户体验上来后会影响效率。不是上线阻塞项。 |
| P3 可选长期 | 做了更好，但当前投入产出比不高，适合后续迭代。 |

---

## 2. P0 立即修复

| 内容 | 原因 | 验收标准 |
|---|---|---|
| 审查报告/服务器资料脱敏，移出项目目录，并补 `.gitignore` 规则 | 当前项目目录中存在含敏感信息的审查/服务器资料类文件，误 `git add .` 风险极高。只移出一次不够，必须防止以后复发。 | 项目根目录不再保存明文凭据文档；`.gitignore` 已忽略 `*审查报告*`、`*服务器*`、`*凭据*`、`.env`、`*.key`、`*.pem` 等敏感文件类型；`git status` 不再出现这些文件。 |
| 轮换已暴露的 GitHub Token / API Key | Token 和 API Key 一旦被复制或误提交，可直接访问代码仓库或产生调用费用。 | 旧 Token/API Key 已吊销；新密钥只保存在安全位置，不进入项目目录和 Git。 |
| 修复服务器备份任务，并做恢复验证 | 服务器有备份 cron，但实际日志显示备份失败；没有可用备份，生产数据没有兜底。 | 手动执行备份成功；`/opt/backups/paper-mes` 生成当天数据库备份和上传目录备份；至少完成一次测试恢复演练。 |
| 登录接口限流 | 系统已公网开放，`/api/auth/login` 没有限流，容易被暴力尝试。 | 只对 `mes.nbsmzwl.cn` 的 `/api/auth/login` 配置 Nginx 限流；不影响服务器上的 WMS/ERP/JimuReport；`nginx -t` 通过并 reload。 |
| 默认账号初始化改造 | 代码仍会在空库生成弱默认密码，新环境或重建数据库会复现风险。 | 默认账号不再使用固定弱密码；改为环境变量指定或启动时随机生成；登录页和设计文档不再提示默认账号。 |
| 统一密码复杂度规则 | 当前初始密码、管理员重置、个人改密都只要求 6 位，生产偏弱。 | 后端 DTO 和前端表单统一校验；至少 8 位，建议要求字母和数字；个人改密和管理员重置规则一致。 |

## 3. P1 投产必做

建议先做低成本、马上改善体验和排查的问题，再做结构治理。

| 顺序 | 内容 | 原因 | 验收标准 |
|---|---|---|---|
| 1 | 新增 404 页面 | 成本很低，避免错误路径白屏，线上体验更稳。 | 不存在路径显示友好页面，提供返回首页/上一页入口。 |
| 2 | 统一前端错误提示策略 | 当前容易出现重复 toast、未捕获 `BizError`、用户只看到“服务器内部错误”。 | 同一错误只提示一次；业务错误有明确文案；控制台不再出现未处理 Promise。 |
| 3 | 数据库迁移治理 | 防止本地代码和生产数据库结构不一致，减少生产 500。 | 建立 Flyway/Liquibase 或 schema version 检查；后续表结构变化必须脚本化并随部署执行。 |
| 4 | 修复 `biz_process_config_draft.version` 默认值 | 线上确认默认值仍是 0，和其他表默认 1 不一致，修复成本低。 | 新迁移脚本将默认值改为 1；历史 `version=0` 数据经过评估后修正。 |
| 5 | `/files/**` 上传文件鉴权 | 上传文件可能包含业务资料；当前资源映射本身无鉴权，知道 URL 即可访问。 | 文件访问经过登录校验；未登录访问返回 401 或跳转登录；已上传文件仍可正常查看。 |
| 6 | 增加 logrotate | 避免 `/var/log/paper-mes` 日志长期增长。 | `/etc/logrotate.d/paper-mes` 生效；日志按天或大小切割，保留 14-30 天。 |
| 7 | 补核心 Controller/集成测试 | 减少“本地正常、生产某入口 500”的回归。 | 覆盖客户/纸张/机台/仓库新增、加工单创建、回录、出库、结算等核心接口。 |
| 8 | CSP 先用 Report-Only | 增强 XSS 防护，但直接阻断可能影响生产资源加载。 | 先配置 `Content-Security-Policy-Report-Only` 观察违规，再逐步切换到正式 CSP。 |

## 4. P2 近期优化

| 内容 | 原因 | 建议时机 |
|---|---|---|
| 优化出库 `DeliveryServiceImpl` N+1 查询 | 出库明细多后会慢，报告中指出的循环查询真实存在。 | P0/P1 完成后，结合出库实际使用频率处理。 |
| 批量更新优化 | 回录、出库、费用计算部分存在循环 `updateById`。 | 数据量上来前逐步优化热点路径。 |
| 自动草稿保存 | 加工单创建步骤多，刷新、过期或误关闭可能丢进度。 | 创建加工单流程稳定后实现定时保存或本地草稿。 |
| 仪表盘/列表 Skeleton 补齐 | 部分详情/表单页已有 Skeleton，但仪表盘和复杂列表仍可改善等待体验。 | 有空做 UI polish 时处理。 |
| 路线设计器撤销/重做 | 路线编辑复杂，误操作恢复成本高。 | 路线设计器继续增强时做。 |
| 慢查询驱动补索引 | 不建议盲目加索引，避免增加写入成本。 | 开启慢查询或用 EXPLAIN 确认后再加。 |
| 基础档案子菜单 icon 一致性 | 视觉轻微不一致，事实存在但影响小。 | 作为 P2/P3 体验优化顺手做。 |

## 5. P3 可选长期

| 内容 | 原因 |
|---|---|
| httpOnly Cookie 替代 localStorage token | 更安全，但涉及登录、登出、CSRF、请求拦截等整套改造；先做 CSP、限流、密码策略。 |
| 选择性外键 / CHECK 约束 | 数据库层更稳，但生产库直接全量加约束风险高；应先做数据巡检和迁移治理。 |
| 渐进拆分 `ProcessOrderServiceImpl` | 类确实偏大，但不是当前生产阻塞；适合随业务改动逐步拆。 |
| i18n 国际化 | 当前中文 MES 场景不需要。 |
| 暗色模式 | 系统确实不支持，但对当前 MES 投产价值低。 |
| API 版本化 | 暂无外部 API 消费方。 |
| 微服务/水平扩展 | 当前单体 Spring Boot 对个人开发和单机部署足够。 |

---

## 6. 推荐执行顺序

1. **清理凭据和 Git 风险**
   脱敏/移出审查报告和服务器资料，补 `.gitignore`，轮换已暴露 Token/API Key。

2. **修复生产备份**
   修 cron/env 加载问题，手动跑通备份，验证恢复。

3. **登录与密码安全**
   配置登录限流，改默认账号初始化，统一密码强度规则。

4. **先做两个低成本体验修复**
   404 页面、统一错误提示策略。

5. **数据库迁移治理**
   建立迁移版本机制，修 `biz_process_config_draft.version` 默认值。

6. **补核心测试和文件鉴权**
   覆盖核心接口链路，收紧 `/files/**` 访问。

7. **再做性能和 UX 优化**
   N+1、批量更新、自动草稿、Skeleton、路线撤销重做。

---

## 7. 当前不建议做的事

1. 不建议上线前大拆 `ProcessOrderServiceImpl`。
2. 不建议直接给生产库全量加外键。
3. 不建议马上重构为 httpOnly Cookie。
4. 不建议投入 i18n、暗色模式、微服务等低收益项。
5. 不建议全局修改 Nginx 限流或安全策略，避免影响同服务器其他生产项目。

---

## 8. 一句话结论

当前真正要先补的是：**凭据不进 Git、备份可恢复、登录口抗暴力、默认密码不可复现、数据库结构可迁移、错误可诊断**。这些完成后，再做性能和 UI/UX 优化，系统才算从“能跑”进入“可放心维护”。
