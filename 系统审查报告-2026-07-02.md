## 一、加工模块（含多工序）

### 1.1 做得好的 ✓

- **创建加工单**：`create()` 事务完整，自动为每条非直发原纸生成第一道主工序（`createMainStepIfNeeded()`），直发卷正确跳过；客户默认结算方式/税率/开票自动带入；`validateMainStepType()` 强制标准加工/现场定尺必填主工艺。
- **卷号并发安全**：`RollNoSequenceService`→`DocumentNoService` 用 `SELECT ... FOR UPDATE` 行锁原子自增 + 唯一索引 `uk_finish_roll_no` + `DuplicateKeyException` 5次重试，三重防护，多账号同时开单不会重号。
- **打印下发**：首打前置校验完整（每卷必须有主工序+正式成品号才可打印），补打强制填写原因留痕，`snap_print` 快照强制 `schema_version` 双重校验。
- **回录闭合校验按母卷分组**：`computeClosureChecks()` 以原纸 uuid 建桶，直发卷正确排除出闭合基准，成品实重/报废/修边按 `biz_finish_original_rel.share_ratio` 分摊归集，符合整改计划 P2-1"单卷闭合"要求。
- **超差放行留痕**：BLOCK 级必须授权+原因，写入操作日志（卷号、偏差率、原因），符合文档 5.7 强制要求。
- **WeightCheckCalculator**：阈值边界（≤2%/2~5%/>5%，含等号）处理正确，`actual=0` 除零防护到位（0/非0→强制BLOCK，0/0→PASS），单测覆盖各档位。
- **RewindWeightCalculator**：5类复卷模式、修边分摊、末件尾差倒挤、实称优先全部实现，单测 6 个用例直接对标规格书 §F，且每个用例断言整卷严格闭合——这是全系统质量最高的一块代码。
- **状态机**：`OrderStatus.allowedTargets()` 精确定义流转矩阵（含两处回退），回退前置校验完整（未结算+未被结算单引用+无成品已出库），回退时数据清理区分两种场景分别处理，逻辑严谨。
- **卷号作废**：仅"预生成"状态可作废，作废永久封存无恢复接口，批量作废先全量校验再逐条更新且乐观锁断言，符合"作废封存不复用"。
- **草稿箱**：草稿提交复用与正式单相同的 `saveFinishConfig()`，核心生成逻辑不分裂。

### 1.2 高严重度问题 🔴

**① 工序损耗无写入路径，闭合校验的"损耗"分量恒为0**
- `ProcessStep.lossWeight` 在整个 processorder 模块中只有实体字段声明，全仓库搜索 `setLossWeight` 仅命中出库模块的快照读取展示，**没有任何业务代码为它赋值**；`BackRecordDTO`/`BackRecordRollDTO`/`BackRecordFinishDTO` 均无损耗输入字段。
- 后果：三级闭合公式"原纸实重 ≈ Σ成品实重 + **Σ工序损耗** + Σ报废 + Σ修边"中第二项永远是0，闭合校验退化为三项比对；文档 2.7"各工序损耗重量自动汇总参与闭合"未落地；损耗全部被挤压到"报废+修边+偏差"里，损耗分析报表（见报表模块）随之失真。
- 同理 `ProcessStageOutput.actualWeight`（多工序中间产出实际重量）也无写入路径，中间工序的实际产出/损耗完全不可追溯。

**② 卷号格式约束缺失**
- 文档规定"1位大写字母（排除I/O/L/Z）+6位数字，满999999自动进位下一字母"。实际实现：`SystemNoRuleBootstrap` 只是种子配置了 `prefix="A"`、`serialLength=6`（运行时可被后台随意改），`NoRuleServiceImpl.PREFIX_PATTERN` 允许任意字母数字前缀，`DocumentNoService.nextSequence()` **无溢出拦截、无字母进位、无 I/O/L/Z 排除逻辑**。流水号超 999999 后会直接产生 7 位数字卷号。
- 短期业务量可能到不了百万级，但这属于"业务规格与代码的直接冲突"，且后台可随意改前缀会破坏"全局唯一卷号体系"的严肃性。

**③ 多工序功能：设计意图与实现存在实质差距（重点）**
系统里实际存在**两套并行的多工序机制**：

| | 机制A：追加工序 `/steps` | 机制B：工艺路线 `/route-config` |
|---|---|---|
| 数据模型 | 扁平 `biz_process_step`，`is_main` 区分 | `stageLevel`/`inputType`/`inputOutputUuid` + `biz_process_stage_output` 中间产出表 |
| 可操作状态 | 待下发(1)可全量维护；待回录(3)仅可加追加工序 | **仅待下发(1)**，且整单全量替换 |
| 链式依赖 | 无（不能引用上一道产出） | 有（第二道输入=第一道产出） |

问题：
- **"先得到第一道产出规格与预估重量，才能配第二道"这一约束只存在于前端交互引导**（`RouteStageToolbar` 按钮 `disabled={!firstOutputs.length}`），后端 `ProcessRoutePreviewer` 接受一次性提交完整 stages 数组，无分步持久化约束——绕过前端即可跳过该规则。
- **回录后无法再配多工序**：路由机制强制 PENDING 状态，而车间实际是"加工中临时发现要追加工序、回录时录入"（文档 2.7），这条真实场景只能走机制A的扁平追加，但机制A不支持"第二道输入=第一道产出"的链式建模。两套机制覆盖不同时间窗，概念相近却无互斥保护和使用引导，集成方极易混淆。
- **多输入阶段血缘丢失**：`ProcessRouteStepWriter.firstInputOutput()` 只取 `inputOutputKeys.get(0)` 记录父子关系，多个中间产出合并进下一道时，除第一个外的父产出关联全部丢失（重量计算正确，但溯源链断裂）。
- **局部调整能力缺失**：路由只支持全量替换（有防呆：已有回录实重禁止重配），任何微调都要重提完整数组。
- 计费叠加本身正确：`calcFee()` 按 `originalUuid` merge 累加多道工序费用，路由写入的工序也会被统一纳入计费。但 `ProcessRouteFeeCalculator` 类写了却无任何调用方，疑似半成品死代码。

### 1.3 中低严重度问题 ⚠

- **`is_mix_process` 双重计算互相覆盖**（中）：`updateMixProcessFlag()`（同卷多工序也算混合）与 `calcFee()`（仅锯纸+复卷并存才算混合）两套判定，后者总在其后执行并覆盖前者。"同卷两道锯纸"场景最终标记会被重置为非混合，与多工序功能冲突。
- **复卷预览与保存两套算法**（中）：`buildRewindSaveSpecs()` 在前端按 segments 提交时走内联分摊逻辑（无实称优先），不走 `RewindWeightCalculator`；预览显示与实际落库重量在实称等边界场景可能不一致——这正是整改计划反复强调"前端不得假计算"后仍残留的口径分裂。
- **草稿预览与提交路径独立**（中）：草稿预览走 `ProcessPlanDraftManager` 四分支，提交走 `saveFinishConfig()` 重算，无契约测试锁定两者一致性。
- 面积公式在 `ProcessOrderServiceImpl` 与 `RewindWeightCalculator` 重复实现（低）；卷号重试模板三处重复（低）；`readConfig()` 靠 JSON 字段特征猜测 DTO 类型，脆弱（低）。

### 1.4 能做什么 / 不能做什么

**能做**：混合工艺单（同单锯纸+复卷）、5类复卷模式全量计算、现场定尺预生成+批量作废、直发原纸自动产出（source_type=2 沿用母卷号）、按母卷分组三级闭合校验、超差授权放行留痕、双快照对比、状态回退（含数据清理）、草稿渐进保存、下发前配置链式多工序路线。

**不能做**：回录阶段动态调整多工序路线；追踪多工序中间产出实重与分工序损耗；保证卷号超容量后仍符合格式；保证预览重量与落库重量在全部路径一致；正确标记"同卷多道同类工序"的混合标识；局部修改已配置的工艺路线。

## 二、结算模块

### 2.1 做得好的 ✓

- **重复结算三层防护**：应用层查重 + DB 唯一约束（生成列 `order_uuid_active`）+ `SELECT FOR UPDATE` 行锁序列化并发，非常扎实。
- **金额从工序表真实聚合**：`buildDetail()` 从 `biz_process_step` 按 `stepType` 分别累加锯纸/复卷费，不依赖缓存字段，符合"分工艺独立统计"。
- **每工序独立取整后累加**：`FeeCalculator.stepAmount()` 单工序取整到元，结算侧不二次取整；`总额 = 不含税 + 税额` 数学上严格相等，无账不平问题。
- **税率快照优先**：优先取加工单创建时快照的税率，客户档案仅作回退——正确的快照设计。
- **超收拦截确实存在**：`receive()` 显式校验"收款金额超过未收金额，禁止超收"，且已结清不可再收、金额必须>0；行锁+乐观锁双防并发收款，两人同时登记不会双双通过。
- **收款撤销与结算作废**：撤销记原因留痕后重算状态；有有效收款时禁止作废结算单；作废时关联加工单 CAS 精确回退（5→4），明细软删自动释放唯一约束，可重新结算。
- 月结口径靠状态机自然防重：已按单结算的单状态变5，月结查询（限定状态4）天然排除，不会重复计入。

### 2.2 问题 ⚠

- **加工单总额与结算单金额取整口径不一致**（中）：加工单 `total_amount` 取整到元（scale 0），结算明细 `orderAmount` 保留到分（scale 2），两处独立计算，含税场景可差 ±0.5 元，财务对账易起疑。建议统一口径或以一方为准。
- **月结账期以 `order_date`（制单日）为准**（中）：6月制单、8月才完成的单，会在8月以后再跑6月账期时被"补捞"进来——已关账月份金额事后可能增长，且系统无"账期锁定"机制。需与业务确认是否可接受。
- **零金额结算单永远卡在"待结算"**（低）：`totalAmount=0` 时收款入口直接拒绝，而状态重算只在收款/撤销时触发，无"总额为0直接置已结清"分支。
- 撤销是软撤销非会计红冲（低，需在使用说明中明确）；结算作废与收款登记共用权限无审批流（低）。

### 2.3 能做什么 / 不能做什么

**能做**：按单/勾选多单/按月三种方式结算；锯纸复卷费分开统计；含税换算；多次收款+撤销；结算作废并释放重结；并发安全。

**不能做**：对非"已完成"单结算；重复结算；跨客户合并结算；超收；有收款时作废；真正的红冲；账期锁定。

## 三、出库模块

### 3.1 做得好的 ✓

- **同一成品防重复勾选**：`lockFinishRolls()` 行锁 + `uk_biz_delivery_detail_active_finish` 唯一索引双重防护，且锁定前排序防死锁——教科书级实现。
- **确认出库 CAS 流转**：逐件 `eq(finishStatus,2).set(finishStatus,3)` 条件更新+乐观锁，任何并发修改导致整单回滚。
- **出库回退**：已出库可回退（3→2），但已被结算引用的禁止回退，闭环正确。
- **跨客户拦截**：创建时逐件校验成品归属客户。
- **直发原纸**：回录自动生成、沿用母卷号、跳过待入库直接可出库、出库列表标识"直发原纸"来源，方向正确。
- **出库结算解耦**：月结客户先出库后对账全程通畅，正向出库不依赖任何结算前置。

### 3.2 高严重度问题 🔴

**① 次结拦截存在根本性漏判**（`DeliveryServiceImpl.hasUnsettledCash()` 第1318-1324行）
- 判定逻辑 = "该客户名下存在状态为待结算/部分收款的结算单"。致命漏洞：**客户的加工单已完成但从未生成过结算单时，查询结果为0，判定为'无欠款'，完全不拦截**——恰恰是风险最大的"没开账单就提货"场景漏掉了。
- 次生问题：口径是客户全量而非本次出库关联单据，客户有一张无关历史欠款（哪怕月结产生）也会误拦本次已结清的出库。
- 建议口径改为：本次出库勾选成品所属的次结加工单，逐单检查"已生成结算单且已结清"，未生成结算单视同未结清。

**② 强制拦截档位与配置开关未实现**
- `BLOCK_REJECT=2` 常量声明后从未被使用；`sys_config_item` 中无出库拦截相关配置键。当前是写死的"默认拦截+`forceRelease=true` 即可放行"，且放行无需任何额外权限/审批（有出库权限即可绕过），文档所述"系统配置弹窗警告/强制拦截二档"不存在。

**③ 分批出库未实现，部分出库重量"蒸发"**
- `remaining_weight` 字段在实体、DDL、全部代码中均不存在（Phase 5 计划里的 DDL 从未执行）。`validateOutWeight()` 允许填部分重量（0<出库重≤实重），但确认后成品**整体**置为已出库，从可出库列表永久消失——物理上未提走的差额无法追踪、无法二次出库。
- 连带违反直发"三不约束"：直发卷（不允许拆分）同样可以填部分重量出库，代码层无 `source_type=2 必须整卷出库` 的校验。

### 3.3 其他问题 ⚠

- 创建后确认前不再重查欠款，存在"创建时无欠款、确认前新欠款"时间窗（低）。
- 明细清空后的空壳待出库单无整单作废/删除接口（低）。
- `pendingDeliveryFinishUuids()` 每次全表扫待出库明细判占用，数据量增长后有性能隐患（低）。

### 3.4 能做什么 / 不能做什么

**能做**：按客户勾选已入库成品（含直发）创建出库单、多加工单合并出库、追加/移除明细、确认扣库存、出库回退、次结欠款（有结算单时）拦截+放行留痕、并发防重。

**不能做**：分批出库剩余量追踪；拦截"从未开结算单"的次结提货；配置化的拦截档位；直发整卷强制出库；空壳出库单作废。

## 四、报表模块

### 4.1 做得好的 ✓

- **无 SUM 放大**：所有报表统一"子查询预聚合再 LEFT JOIN"模式，order→roll→step 多层关联不会放大金额——代码质量高。
- 软删过滤全覆盖；除零全部有 CASE WHEN 防护；备用号/作废号（`is_spare`、`roll_no_status=3`）正确排除。
- 收款用流水表按有效状态聚合，非静态字段，口径正确；应收对已结算单取结算分摊、未结算单退化取订单总额，设计合理。
- 前端千分位/吨位（kg÷1000 三位小数）/百分比三端（表格/看板/导出Excel）格式统一；`/api/reports/export` 三 Sheet 导出完整且与页面口径一致。

### 4.2 数据口径风险（重点）

| 口径 | 现状 | 严重度 |
|------|------|--------|
| 损耗数据源 | 报表读 `biz_original_roll.total_loss_weight/ratio`，该字段**仅种子SQL写入、应用代码从不回写**（与加工模块问题①同源），损耗报表实际长期为0或陈旧值；也无水湿/破损/修边分类维度 | 🔴 高 |
| 直发原纸 | 原纸吨位/损耗聚合未剔除 `process_mode=3`，直发（未加工）重量虚增加工吨位、拉低表观损耗率；回录闭合逻辑却是剔除的，两处口径打架 | 🟠 中高 |
| 报废成品 | 成品聚合未排除 `finish_status=4`，报废卷实重计入"成品吨位"，与看板"成品库存"（严格 status=2）数字对不上 | 🟡 中 |
| 时间口径 | 全部按 `order_date`（制单日），前端筛选器未标注，跨月完工单会造成"月报表≠月实际产出"的解读偏差 | 🟡 中 |
| 机台归因 | `machine_uuid` 只在原纸维度，多工序跨机台时无法按工序拆分产出 | 🟡 中（结构性） |
| 影子接口 | `/monthly` `/customer` `/loss` `/machine` 四个端点+前端4个Hook已定义但页面从未调用（页面统一走 `/dimensions`），且其状态过滤硬编码与主口径不一致，未来被误用会导致口径分裂 | 🟡 中 |
| SQL重复 | `SettleAllocationByOrder` 约35行片段在 ReportMapper.xml 与 DashboardMapper.xml 逐字重复，漂移风险 | 🟡 中 |
| 索引 | `biz_original_roll.machine_uuid` 无索引，机台维度大数据量下慢查询 | 🟢 低中 |

### 4.3 能做什么 / 不能做什么

**能做**：按制单日期任意区间的月度/客户/维度汇总与明细，金额可拆应收/加工费/附加费/已收/未收，Excel 导出，看板与报表主口径基本一致。

**不能做**：可信的损耗分析（数据源断供）；损耗分类（水湿/破损/修边）；按完工日期统计；多工序跨机台精确归因；剔除直发/报废的净加工口径。

## 五、前端 UI/UX

### 5.1 做得好的 ✓

- **状态机驱动的单一主动作**：列表行与详情页按订单状态只暴露唯一"下一步"按钮（继续编辑→打印下发→转待回录→进入回录→创建出库），对操作水平一般的用户非常友好。
- **多页签体验**：类浏览器 PageTabs（右键关闭当前/左侧/右侧/其他），适合办公室多单并行。
- **回录工作台**：三栏布局（母卷导航|录入面板|实时闭合预览）、偏差分级变色、缺口标签（"母卷复称未填/成品X项未填"）、"整单按理论回录"一键填充——整体是全前端体验最好的页面。
- **多工序引导**：追加锯纸/复卷按钮在无首道产出时禁用+顶部 Alert 双重提示；直发卷正确隐藏配置表单，不被多工序 UI 干扰；锯纸门幅可视化进度条（超宽变红、剩余转切边一键操作）直观。
- **统一错误层**：`api/request.ts` 错误码映射中文、401 跳登录带回跳、超时/网络错误区分文案；空态覆盖 25 个文件。
- 状态色单一数据源（`constants/processOrder.ts`）、列宽/显隐 localStorage 持久化、删除均有说明性二次确认、权限 403 页面友好。

### 5.2 问题 ⚠

**🔴 高**
1. **状态色标偏离设计文档**：待下发应为橙实为灰（default）、待回录应为红实为橙（warning）、已结算应为灰实为青（cyan），5个状态错3个。待回录本应是最醒目的红色警示。好在色源单一，改一处全局生效。
2. **出库/结算列表页无按钮级权限控制**：签收/回退/登记收款等高敏按钮只靠路由级 view 权限，有查看权即可见（点了才报错），与客户等模块的 `useHasPermission` 收敛策略不一致。
3. **组件重复与死代码**：两个同名 `PlanPreviewPanel`（精度 2位 vs 3位、能力不同）并存；5 个 Drawer 组件（OrderCreateDrawer/OrderDetailDrawer/BackRecordDrawer/DeliveryCreateDrawer/DeliveryDetailDrawer）已被页面化路由取代但未清理，误导后续开发。

**🟡 中**
4. 重量精度混用：18 个文件 toFixed(3)、11 个文件 toFixed(2)，同一组件内同一"吨"字段两种精度（ProductionFlowTable 111/185行），不同页面合计视觉对不上；建议统一 formatter（重量3位/金额2位）。
5. 基础档案五个列表页（客户/纸张/机台/仓库/用户）完全没有导出功能；导出文件名用时间戳而非单号+日期，难以区分。
6. 回录工作台无键盘流（无 Enter 跳下一项、无快捷切换母卷），高频数字录入效率受限；"下一项"到末尾静默循环回第一项无提示。
7. "倒挤尾差"业务术语未在回录 UI 呈现（仅有"估算差额"），办公室与车间/财务口径对不上。
8. 单工序/直发订单也进入固定三栏多工序工作台，无轻量路径，对简单单据有认知负担。
9. 多道工艺追加 Drawer 纵向堆叠无折叠，3道以上视觉密度过高。
10. 出库导出仅支持单选单张，不支持按筛选条件批量导出对账。

**🟢 低**
- 部分状态回退按钮无 loading 态可连点；无 `navigator.onLine` 离线感知；无虚拟滚动（当前量级可接受）；`ConfigFinishPage` 等长文件大量内联 style；无 aria-label（状态 Tag 有文字，色弱可辨性尚可）。

### 6.2 已完善的能力

主链路闭环、并发安全体系（行锁+乐观锁+唯一索引+ConcurrencyGuard 断言）、状态机与回退、双快照与超差留痕、5类复卷计算引擎（含规格书对标单测）、按母卷分组闭合校验、结算防重与超收拦截、报表防 SUM 放大的预聚合架构、前端状态驱动交互与回录工作台。

### 6.3 主要缺口（按优先级）

**P0（影响资金/数据正确性，建议立即处理）**
1. 次结拦截漏判：改为按本次出库关联加工单逐单判定，未生成结算单视同未结清（DeliveryServiceImpl.hasUnsettledCash）
2. 工序损耗写入链路：回录 DTO 增加损耗字段→写 `ProcessStep.lossWeight`→回写 `biz_original_roll.total_loss_weight/ratio`，打通闭合校验与损耗报表
3. 报表口径三连修：原纸/损耗聚合剔除 `process_mode=3`；成品聚合排除 `finish_status=4`；损耗报表改从工序+报废+修边实时聚合

**P1（影响业务完整性）**
4. 分批出库：落地 remaining_weight 方案，或明确告知业务"当前只支持整卷出库"并在前端禁止填部分重量；同时给直发卷加"必须整卷出库"校验
5. 卷号字母进位与 I/O/L/Z 排除的代码级实现（或至少加溢出告警）
6. 多工序：明确两套机制的定位与互斥规则；补"回录阶段基于第一道实际产出追加工序"的能力；修复多输入血缘只记第一个的问题
7. is_mix_process 双逻辑合一；复卷预览/保存统一走 RewindWeightCalculator
8. 状态色标按文档修正（一处改动全局生效）

**P2（体验与工程质量）**
9. 出库/结算按钮级权限；基础档案导出；导出文件名可读化
10. 回录键盘流 + "倒挤尾差"术语呈现；单工序轻量配置路径；多道工艺折叠
11. 清理死代码（5个Drawer、4个影子报表端点、ProcessRouteFeeCalculator、重复PlanPreviewPanel）；统一重量/金额 formatter；SettleAllocationByOrder SQL 去重；补 machine_uuid 索引
12. 零金额结算单自动结清；账期锁定机制评估；加工单/结算单金额取整口径统一

### 6.4 交互体验最需优化的三件事

1. **让"待回录"红起来**——车间单据回来了没录，是这套系统最怕积压的环节，颜色必须一眼可辨。
2. **回录键盘流**——回录是打字量最大的页面，Enter 跳格、快捷切母卷能把每单录入时间明显压下来。
3. **多工序对简单单据零打扰**——直发/单工序单给极简确认卡片，检测到需要多工序再展开三栏工作台，别让每个制单员都面对最复杂的界面。
